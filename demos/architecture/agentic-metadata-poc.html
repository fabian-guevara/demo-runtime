MongoDB Atlas architecture

Agentic Metadata Planner

This demo positions MongoDB Atlas as the application-facing retrieval and reasoning layer over warehouse-style metadata. Atlas stores table and join metadata, executes Atlas Vector Search to discover relevant entities, and persists every planned query run. SQL generation and narrative refinement happen outside MongoDB; the database owns discovery, relationship context, and auditability.

When metadata becomes the bottleneck

Business users ask warehouse questions every day. Finding the right tables, join paths, and safe SQL still depends on analysts, stale wikis, and tribal knowledge. Atlas can sit in front of the warehouse as the retrieval and audit layer—without replacing Snowflake, Databricks, or Redshift as the engine of record.

Situation Why it hurts Where Atlas fits In the demo
Schema discovery is slow Hundreds of tables and columns; business users depend on analysts or stale wikis to find the right assets. table_nodes store business descriptions, tags, and sample questions; Atlas Vector Search retrieves candidates from natural language. Ask a business question → relevant tables appear with similarity scores in the UI.
Join logic is tribal knowledge Relationship rules live in people’s heads, not in a searchable system agents can use safely. table_edges persist join metadata; GraphRAG stores entities in metadata_knowledge_graph and traverses joins with Atlas $graphLookup. Retrieved edges show how customers, plans, billing, and migrations connect.
Agents write unsafe SQL LLMs guess table names and joins, producing wrong or unrunnable warehouse queries. Deterministic planner assembles SQL from retrieved metadata first; the LLM ({{LLM_MODEL}}) only refines the explanation. Generated SQL maps to known join keys from MongoDB metadata, not free-form model invention.
No replay or governance Teams cannot answer “what tables and joins did the agent use for this question?” query_runs persists full retrieval context, plan, SQL, explanation, and timings. MongoDB terminal shows vector searches, GraphRAG traversal, and the saved run payload.
Warehouse vs. agent gap Leadership wants one conversational layer over Snowflake/Databricks, but metadata tools are separate from AI chat. MongoDB holds the agentic retrieval and planning layer; warehouse remains the analytics engine of record. Mongo alternative rationale explains how the same intent could be modeled operationally in MongoDB.

Index strategy on Atlas

B-tree indexes (created by Seed)

Collection Index Purpose
table_nodes { tableName: 1 } unique
{ schemaName: 1 }		 Catalog lookup and deterministic joins during planning
table_edges { sourceTable: 1, targetTable: 1 } Edge lookup during graph expansion
query_runs { createdAt: -1 } Recent run history

Atlas Vector Search indexes (manual Atlas setup)

Index name Collection Vector field Model
table_nodes_vector_index table_nodes embedding
1024 dims, cosine		 Client-side Voyage at seed + query time
table_edges_vector_index table_edges embedding
1024 dims, cosine		 Client-side Voyage at seed + query time

Filter fields on both vector indexes: schemaName, tableName on nodes; sourceTable, targetTable on edges. This keeps pre-filtering inside Atlas Vector Search rather than post-filtering large candidate sets in application code.

GraphRAG pattern (MongoDB Atlas + LangChain-compatible)

This demo implements the GraphRAG pattern that MongoDB Atlas documents with LangChain's MongoDBGraphStore (GA May 2025). The official Python package is langchain-mongodb; here we use the same entity schema and $graphLookup traversal in Node.js with an LLM ({{LLM_MODEL}}) for entity extraction and explanation. 

{
  $graphLookup: {
    from: "metadata_knowledge_graph",
    startWith: "$relationships.target_ids",
    connectFromField: "relationships.target_ids",
    connectToField: "_id",
    as: "connections",
    maxDepth: 3
  }
}

Vector Search pattern in this demo

Metadata retrieval here uses bring-your-own-vector: Voyage generates embeddings at seed time and again for each question. Atlas stores vectors on the document and searches with queryVector. 

{
  $vectorSearch: {
    index: "table_nodes_vector_index",
    path: "embedding",
    queryVector: <1024-dim question embedding>,
    numCandidates: 40,
    limit: 5
  }
}

Fallback behavior

  • Missing Voyage API key: seed stores embedding: null; retrieval falls back to lexical scoring over the in-memory catalog.
  • Missing Atlas vector indexes: vector search throws; demo falls back to lexical retrieval so the story still runs locally.
  • GraphRAG traversal: seed builds metadata_knowledge_graph; query uses entity extraction plus Atlas $graphLookup (same model as LangChain MongoDBGraphStore).

Typical story mapped to Atlas operations

Question: “How many high-value customers had billing issues after a plan migration in the last 30 days?”

  1. Vector search retrieves customers, customer_value_segments, plan_migrations, billing_events table nodes plus relevant join edges.
  2. GraphRAG traverses connected table entities in metadata_knowledge_graph to validate join paths across customer domains.
  3. Deterministic planner selects intent billing_issue_after_plan_migration and assembles SQL from known join keys.
  4. insertOne on query_runs captures retrieved tables, edges, SQL, explanation, and latency breakdown.

Operational checklist on Atlas

  • Cluster: MongoDB Atlas, same region as app for predictable Vector Search latency.
  • Database: agentic_metadata_demo (override with MONGODB_DB).
  • Run Seed from demo runtime to load metadata and create B-tree indexes.
  • Create both Vector Search indexes in Atlas Search before expecting semantic retrieval.
  • Set VOYAGE_API_KEY if you want true vector mode; otherwise lexical fallback applies.
