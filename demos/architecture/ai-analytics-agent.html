MongoDB Atlas architecture

AI Analytics Agent

This demo uses MongoDB Atlas as the operational data plane, analytics engine, semantic retrieval layer, and agent memory store for a telecom-style leadership analytics assistant. Structured churn analytics run through the Aggregation Framework. Evidence retrieval uses Atlas Vector Search with client-side Voyage embeddings. When Voyage is unavailable the app uses deterministic mock_embedding_local_only vectors for local smoke tests, then may fall back to lexical_degraded token scoring if vector search fails. Follow-up questions and leadership preferences are persisted in MongoDB-backed short- and long-term memory collections.

When leadership needs answers, not another export

Churn, revenue at risk, NPS, and operational incidents show up in ad hoc questions and follow-ups—not just scheduled dashboards. Atlas can hold the analytics, evidence, memory, and audit trail in one place while The LLM (Grove, default gpt-5.5) turns verified facts into executive language.

Seed embeds documents with Voyage when VOYAGE_API_KEY is set. At query time the app embeds the question and runs $vectorSearch with queryVector.

Local-only warning: without VOYAGE_API_KEY, the app uses deterministic mock_embedding_local_only embeddings. That mode is for local smoke tests only — it is not real semantic retrieval. If vector search still fails or returns no matches, evidence tools fall back to lexical_degraded token scoring.

Conversation flow through Atlas

Facts always come from Atlas first. Grove (default gpt-5.5) only narrates structured JSON the demo API already fetched—it never invents metrics or reads MongoDB directly.

Turn 1 — churn question

  1. User → Demo API Churn question
  2. Demo API → Atlas findOne on agent_checkpoints
  3. Demo API → Atlas aggregate churnRiskPipeline
  4. Demo API → Atlas aggregate $vectorSearch on support + incident evidence
  5. Atlas → Demo API KPIs, high-risk accounts, and evidence docs
  6. Demo API → Atlas updateOne checkpoint + insertOne telemetry
  7. Demo API → User Executive answer

Between steps 5 and 7, the LLM narrates the structured facts returned by Atlas. It does not query MongoDB.

Turn 2 — follow-up comparison

  1. User → Demo API Compare to last month
  2. Demo API → Atlas Read checkpoint context
  3. Demo API → Atlas aggregate comparisonPipeline
  4. Atlas → Demo API Month-over-month facts
  5. Demo API → Atlas updateOne checkpoint
  6. Demo API → User Executive answer

The LLM again narrates facts already materialized by MongoDB aggregations.

Analytics pattern: churn in one aggregation

Leadership metrics are computed inside Atlas using a single aggregation with $facet so summary KPIs and top accounts come from one round trip. 

usage_metrics.aggregate([
  { $match: { region, segment, month } },
  { $lookup: { from: "accounts", localField: "accountId", foreignField: "accountId", as: "account" } },
  { $unwind: "$account" },
  { $match: { "account.status": "active" } },
  { $facet: {
      summary: [ { $group: { avgChurnRiskScore, totalRevenueAtRisk, ... } } ],
      highRiskAccounts: [ { $sort: { churnRiskScore: -1 } }, { $limit: 5 } ]
  }}
])

Semantic evidence pattern: Vector Search + fallback

support_interactions.aggregate([
  { $vectorSearch: {
      index: "support_interactions_vector_index",
      path: "embedding",
      queryVector: <1024-dim question embedding>,
      filter: { region: "Texas", segment: "enterprise" },
      numCandidates: 48,
      limit: 3
  }},
  { $project: { score: { $meta: "vectorSearchScore" }, summary: 1, category: 1 } }
])

Observability inside Atlas

Every tool invocation wraps the Mongo operation in telemetry persistence to demo_telemetry: aggregation pipeline, explain summary, duration, returned count, memory type, embedding model, and retrieval mode (vector, mock_embedding_local_only, or lexical_degraded). This is what powers the runtime “What did we run?” panel.

Three-turn demo story

  1. “Show churn risk for Texas enterprise accounts” → churn aggregation on usage_metrics + evidence vector search on support/incident collections.
  2. “Now compare that to last month” → checkpoint read + comparison aggregation across two month partitions.
  3. “Remember leadership cares most about billing disputes, NPS drops, and network incidents” → upsert into agent_memories, influencing subsequent retrieval and LLM narrative context.

Operational checklist on Atlas

  • Database: telco_ai_analytics_demo (override with MONGODB_DB_NAME).
  • Set MONGODB_URI, GROVE_API_KEY, GROVE_MODEL, GROVE_API_URL, and VOYAGE_API_KEY (plus VOYAGE_EMBEDDING_MODEL / VOYAGE_EMBEDDING_DIMENSIONS).
  • Run Seed to load accounts, metrics, support notes, incidents, and B-tree indexes.
  • Create all three Vector Search indexes in Atlas before semantic retrieval will work.
  • Set VOYAGE_API_KEY for real vector mode; otherwise expect mock_embedding_local_only and possible lexical_degraded fallback.
  • Grove is required for user-facing answers, but all facts come from MongoDB first.
