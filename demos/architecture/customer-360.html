MongoDB Atlas architecture

Customer 360 Command Center

This demo uses MongoDB Atlas as the system of record for 1M+ customer profiles, interaction history, care knowledge, and chat audit. The UI search bar and agent lookup tools run Atlas Search $search against the customers collection. Retention guidance uses Atlas Vector Search on care_kb. The LLM ({{LLM_MODEL}}) orchestrates cohort analytics through the MongoDB MCP Server plus custom lookup and search tools.

When care teams need a unified customer view at scale

Operators cannot grep spreadsheets or run ad hoc regex scans across a million subscribers. They need fast lookup by phone, email, or name, a 360 profile with recent interactions, and agent-assisted retention guidance grounded in MongoDB — not invented account details.

Situation Why it hurts Where Atlas fits In the demo
Fragmented customer data Plan, device, churn score, and care history live in different silos. Single customers collection plus interactions keyed by customerId. Profile panel loads customer + last 12 interactions in one API call.
Lookup at scale Regex scans on 1M rows are slow and do not rank by relevance. Atlas Search index customers_search_index on name, email, msisdn, and customerId. Sidebar search runs $search with segment filter; regex fallback if index unavailable.
Retention playbooks elsewhere Agents improvise offers without policy context. care_kb with Voyage embeddings + Vector Search index care_kb_vector_index. Chat calls vector_search_tool / fusion variants for escalation guidance.
Cohort questions Leadership asks for segment rollups and top at-risk accounts. MCP aggregate / count across indexed fields; precomputed customer_stats. Stats bar + segment_insights_tool for agent-driven summaries.
No proof of what ran Reviews cannot reconstruct which MongoDB queries powered a recommendation. Runtime telemetry tracks tool name, collection, operation, duration, and payload snapshot. Bottom MongoDB terminal + /api/demo/actions poll the same audit stream.

Collections and roles

Collection Domain data Atlas interaction
customers 1M+ profiles: segment, plan, market, churnRisk, LTV, device, spotlight accounts B-tree indexes + Atlas Search index; UI and agent lookup via $search
interactions Care, retail, app, and phone touchpoints linked by customerId find recent history for 360 profile; MCP queries in chat
care_kb Retention and escalation guidance chunks Seed embeds with Voyage; Vector Search via $vectorSearch
customer_stats Precomputed segment rollups after seed Stats bar reads summary doc; refreshed by seed
chat_messages Session history, tool metadata, completion reason Short-term memory before each agent loop

Index strategy on Atlas

B-tree indexes (created by seed)

Collection Index Why it matters
customers { customerId: 1 } unique
{ msisdn: 1 }
{ email: 1 }
{ segment: 1, market: 1 }
{ churnRisk: -1 }		 Exact lookup, segment browse, at-risk sorting when search box is empty
interactions { customerId: 1, occurredAt: -1 }
{ occurredAt: -1 }		 360 profile history and recent activity queries
chat_messages { sessionId: 1, createdAt: -1 } Agent session memory

Atlas Search index — customers

[
  {
    "$search": {
      "index": "customers_search_index",
      "compound": {
        "must": [{
          "text": {
            "query": "Jamie Torres",
            "path": ["searchableName", "firstName", "lastName", "email", "msisdn", "customerId"]
          }
        }],
        "filter": [{
          "equals": { "path": "segment", "value": "postpaid" }
        }]
      }
    }
  },
  { "$limit": 24 },
  {
    "$project": {
      "customerId": 1,
      "firstName": 1,
      "lastName": 1,
      "churnRisk": 1,
      "searchScore": { "$meta": "searchScore" }
    }
  }
]
Index Collection Indexed fields Notes
customers_search_index customers searchableName, firstName, lastName, email, msisdn, customerId, segment, market, plan, status Created/waited on by seed; powers UI search + lookup_customer_tool
care_kb_vector_index care_kb embedding — 1024 dims, cosine Required for retention guidance tools

Existing clusters: run npm run indexes to create the Atlas Search index without reloading 1M customers. Seed backfills searchableName from first + last name before index creation.

Seed strategy

  • Bulk insertMany in 10,000-document batches — default CUSTOMER_SEED_COUNT=1000000.
  • 250k interactions by default via INTERACTION_SEED_COUNT.
  • Spotlight customers at C000000001–C000000004 for deterministic demo flows.
  • B-tree indexes, Atlas Search index, and Vector Search index are created/waited on after load.

Demo checklist

  1. Seed the cluster (npm run reset or hub Seed button).
  2. Search Jamie Torres or 7135550101 — confirm Atlas Search results.
  3. Open the 360 profile and review interactions.
  4. Run the sample retention prompt and inspect MongoDB terminal traces.
