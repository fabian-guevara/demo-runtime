MongoDB Atlas architecture

Tower Network Monitoring

This demo uses MongoDB Atlas as the live telemetry store, tower health rollup, runbook retrieval layer, and chat audit trail for a telecom-style network operations center. Grove (default gpt-5.5) orchestrates reads through the MongoDB MCP Server plus custom search tools backed by Atlas Vector Search with Voyage embeddings. An optional Atlas Charts geospatial embed shows tower status on a map while the simulator keeps realtime_network_logs flowing.

When NOC teams need live context, not another dashboard export

Tower alerts, severity spikes, and remediation steps arrive in bursts. Operators need to query recent logs, identify impacted sites, and pull runbooks without leaving the incident thread. Atlas holds telemetry, health rollups, manuals, and chat history in one cluster while MCP exposes database tools to the agent safely.

Situation Why it hurts Where Atlas fits In the demo
Alert storms High-severity events flood in faster than operators can filter by tower, time window, and severity. Time-series style collection realtime_network_logs with indexes on timestamp, tower, and severity. Start the simulator → batches insert every 5s → MCP find/aggregate on recent logs.
No single tower view Raw logs do not tell you which sites are red on the map or how many open alerts each tower has. Aggregation rollup into tower_health after every log batch; geospatial sites in tower_sites. Status bar + Atlas Charts map read tower_health.mapColor joined to site coordinates.
Runbooks live elsewhere Operators grep PDFs or wikis while the incident clock runs. manuals collection with Voyage embeddings + Atlas Vector Search index manuals_vector_index. Chat calls vector_search_tool (or fusion/text variants) for remediation steps.
Untrusted AI answers Chatbots invent tower IDs or severities without touching the database. Agent loop executes MCP MongoDB tools first; LLM narrates structured tool results. MongoDB terminal shows every request payload and return value behind each answer.
No proof of what ran Post-incident reviews cannot reconstruct which queries powered a recommendation. Runtime telemetry tracks tool name, collection, operation, duration, and response snapshot. Right-side MongoDB terminal + /api/demo/actions poll the same audit stream.

Collections and roles

Collection Domain data Atlas interaction
tower_sites Static tower metadata and GeoJSON points for map charts 2dsphere index on location; read by map API and Atlas Charts

Index strategy on Atlas

B-tree indexes (created by seed)

Collection Index Why it matters
tower_sites { towerId: 1 } unique
{ location: "2dsphere" }		 Map lookups and geospatial Charts
tower_health { towerId: 1 } unique
{ status: 1 }		 Fast status bar and map color joins
realtime_network_logs { event_timestamp: -1 }
{ source_tower_id: 1, event_timestamp: -1 }
{ severity: -1 }		 Recent alert queries and simulator rollups
manuals { alertType: 1 } Runbook filtering during seed and search
chat_messages { sessionId: 1, createdAt: -1 } Short-term chat memory per session

Atlas Vector Search index

Index Collection Vector field Notes
manuals_vector_index manuals embedding
1024 dims, cosine		 Created/waited on by seed; required for vector_search_tool

Embeddings and seed: manual/runbook Vector Search requires VOYAGE_API_KEY. Seed fails fast if embeddings cannot be generated or manuals would be stored without vectors. Atlas model keys (al--...) call https://ai.mongodb.com/v1/embeddings; Voyage platform keys (pa--...) call https://api.voyageai.com/v1/embeddings.

Search tool behavior: text_search_tool uses Atlas Search and falls back to lexical_degraded token scoring when the text index is unavailable. vector_search_tool requires live embeddings and does not silently substitute lexical search.

Agent loop safeguards

The chat service uses a bounded for loop (default 15 steps) with industry-style termination: answer-only mode near the budget, duplicate tool fingerprinting, structured tool errors, and a final synthesis pass if the model never returns {"action":"answer"}.

Tower health rollup

After seed batches and every simulator insert, the API aggregates the last hour of logs per tower and upserts tower_health. Status drives both the in-app status bar and Atlas Charts color encoding. 

realtime_network_logs.aggregate([
  { $match: { event_timestamp: { $gte: lastHour } } },
  { $group: {
      _id: "$source_tower_id",
      maxSeverity: { $max: "$severity" },
      openAlertCount: { $sum: { $cond: [{ $gte: ["$severity", 4] }, 1, 0] } } },
      lastEventAt: { $max: "$event_timestamp" },
      lastDescription: { $last: "$event_description" }
  }}
])
// status: healthy | degraded | critical
// mapColor: green | red

Runbook retrieval pattern

manuals.aggregate([
  { $vectorSearch: {
      index: "manuals_vector_index",
      path: "embedding",
      queryVector: <1024-dim Voyage embedding>,
      numCandidates: 50,
      limit: 5
  }},
  { $project: {
      title: 1,
      alertType: 1,
      probableCauses: 1,
      remediationSteps: 1,
      score: { $meta: "vectorSearchScore" }
  }}
])

MCP vs custom tools

Tool source Examples Purpose
MongoDB MCP Server find, aggregate, collection listing Operator-style database reads against logs, health, and sites
Custom demo tools vector_search_tool, text_search_tool, fusion_search_tool, get_utc_time Runbook retrieval and small utilities not exposed by MCP

Observability

Every MCP and custom tool invocation is wrapped by the runtime tracker. The MongoDB terminal panel polls /api/demo/actions and shows request payloads, return values, duration, and collection metadata. This is the audit trail for “what did the agent actually run?”

Demo story

  1. Seed → load 10 towers, manuals with embeddings, vector index READY, initial logs.
  2. Start simulator → continuous log batches; tower_2 and tower_7 skew toward high severity.
  3. Paste Atlas Charts embed → map reads tower_sites + tower_health.
  4. Run sample incident prompt → MCP reads recent logs, vector search pulls remediation, LLM summarizes impacted towers.
  5. All healthy → API resets tower_health to green for demo reset moments.

Runtime truth table

Concern Runtime behavior
LLM provider Grove (GROVE_API_KEY, default gpt-5.5 via GROVE_MODEL)
Embedding mode Voyage (or Atlas model API key) at seed + query; required for vector search tools
Retrieval mode vector for vector_search_tool; Atlas Search with lexical_degraded fallback for text_search_tool
Required indexes B-tree log/health indexes; Vector Search index manuals_vector_index; optional Atlas Search text index on manuals
Degraded behavior text_search_tool → lexical_degraded; vector_search_tool errors without embeddings
What fails fast Seed without VOYAGE_API_KEY; vector_search_tool when embeddings are not configured

Operational checklist on Atlas

  • Database: network_monitoring (override with MONGODB_DB_NAME).
  • Set MONGODB_URI, GROVE_API_KEY, GROVE_MODEL, GROVE_API_URL, and VOYAGE_API_KEY (plus VOYAGE_EMBEDDING_MODEL / VOYAGE_EMBEDDING_DIMENSIONS).
  • Run npm run seed (or npm run reset) and wait for manuals_vector_index to reach READY.
  • Build the geospatial Charts map per docs/ATLAS_CHARTS.md and paste the embed URL in Config.
  • Ports: API 4003, web UI 5178.
  • Default Grove model is gpt-5.5; facts still come from MongoDB tool results first.
